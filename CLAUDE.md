# CLAUDE.md

## 项目定位

这是一个基于 VitePress 的技术学习文档站，面向有 JavaScript / TypeScript 基础、希望系统学习 Python、Rust、Solidity 的开发者。

写作与修改时优先考虑：

- 从 JavaScript 开发者视角解释概念
- 保持“学习路线 + 知识站”结构，而不是零散笔记
- 强调工程实践、迁移思维和可读性
- 中文为主，必要时保留英文术语
- 尽量让内容适合长期维护，而不是只解决一次性记录需求

## 仓库结构

- `README.md`：项目介绍、开发方式、部署说明
- `docs/index.md`：站点首页
- `docs/.vitepress/config.ts`：VitePress 配置（导航、侧边栏、站点元信息）
- `docs/python/`：Python 学习路线，按 `phase-xx` / `dayxx.md` 组织
- `docs/rust/`：Rust 学习路线，按章节组织
- `docs/solidity/`：Solidity 学习路线，按章节组织

## 本地开发

常用命令：

```bash
npm run dev
npm run build
npm run preview
```

依赖安装：

```bash
npm install
```

## 修改内容时的约束

### 文档内容

- 优先延续现有中文技术写作风格
- 面向“已有 JS 背景的学习者”，不要写成完全零基础教程
- 新内容尽量放在现有目录结构下，不随意新增顶层分类
- 保持标题层级清晰，避免同一篇文档里结构失衡
- 示例应尽量服务于“理解迁移”和“工程实践”，不要只堆语法
- 同一主题下优先补足上下文、迁移说明、实践场景，而不是简单扩写篇幅

### VitePress 结构

- 新增或重命名页面时，同时检查 `docs/.vitepress/config.ts`
- 如果页面需要被导航访问，记得更新 `nav` 或 `sidebar`
- 若修改站点路径、仓库名或部署相关信息，同时检查：
  - `docs/.vitepress/config.ts` 中的 `base`
  - `README.md` 中的仓库名与部署说明
- 不要只改文档文件而忽略导航，否则容易出现页面存在但站内无法访问的情况

### 编辑原则

- 优先小而准的改动，不做无关重构
- 不要批量改写整站文风，除非用户明确要求
- 不要随意移动大量文档路径，否则会影响侧边栏与链接
- 修改 Markdown 时注意内部链接是否仍然有效
- 不要为了“更完整”而加入与当前章节目标无关的大段背景知识

## 写作风格规范

### 语气与表达

- 默认使用清晰、直接、偏教学型表达
- 避免过度口语化，也不要写成论文式表述
- 解释概念时先给“它是什么 / 为什么重要”，再给“怎么用”
- 面向有 JS 经验的读者时，可以适度使用对照表达帮助迁移理解

### 内容组织

优先采用以下组织方式：

1. 概念定义
2. 与 JavaScript / TypeScript 的对照
3. 核心语法或机制说明
4. 示例代码
5. 常见误区 / 迁移陷阱
6. 工程实践建议

不是每篇都必须严格套模板，但新增内容应尽量贴近这种阅读节奏。

### 示例代码

- 示例应短小、聚焦，直接服务于当前知识点
- 不要为了展示技巧引入过多前置知识
- 多语言对照内容里，优先保证“概念对应”而不是语法形式完全对齐
- 如果是安全、异步、工程化主题，示例应体现真实开发场景，而不是玩具例子

### 标题与层级

- `#` 通常只用于页面主标题
- 优先使用 `##` 组织主结构，`###` 组织子主题
- 不要连续出现很多只有一句话的同级标题
- 如果页面已经有固定结构，新增内容应尽量延续原有层级风格

## 命名与文件组织规范

### Python 路线

- 目录使用 `phase-xx-name` 风格
- 每日页面使用 `day01.md`、`day02.md` 这种零填充命名
- phase 目录下通常应包含一个 `index.md` 作为该阶段总览

### Rust / Solidity 路线

- 保持现有 chapter 风格命名，不随意混入新的命名体系
- 速查表、附录、映射类文档应放在对应语言目录下，不要放到顶层

### 新文档落位

新增文档前先判断它属于：

- 某一条学习路线的正式章节
- 某路线的辅助资料
- 站点级页面

不要把“章节内容”误放到“站点首页”或把“辅助资料”混入主线章节列表。

## 适合 Claude 的工作方式

- 先读目标文档和相关导航配置，再动手修改
- 涉及多个页面时，顺手检查是否有目录页/索引页需要同步
- 如果只是补充内容，尽量复用现有版式和段落风格
- 如果要新增一整组文档，优先沿用现有 phase / chapter 风格命名
- 如果用户只要求修正文案、错别字或局部内容，不要顺手扩大改动范围

## 常见任务处理建议

### 新增一篇文档

至少检查：

- 文档放置目录是否正确
- 标题层级是否自然
- 是否需要加入侧边栏或导航
- 是否需要在对应 `index.md` 中补入口或说明
- 是否与现有路线命名风格一致

### 修改某篇已有文档

优先检查：

- 本文是否已有固定结构和术语风格
- 改动是否破坏章节节奏
- 内部链接、相邻章节引用是否仍然成立
- 是否需要同步更新概览页或导航说明

### 修改首页或站点结构

额外检查：

- `docs/index.md`
- `docs/.vitepress/config.ts`
- `README.md`
- 站点 `base`、导航链接、文案定位是否一致

## 提交前建议检查

### 小改动

如果只是非常小的文案修正，可只做基本自检：

- Markdown 结构没有破坏
- 链接没有写错
- 文风与周边内容一致

### 结构性改动

如果本次修改涉及页面结构、导航或多篇文档，建议至少执行：

```bash
npm run build
```

### Claude 自检清单

在完成修改前，优先快速确认：

- 这次改动是否严格围绕用户要求
- 是否误改了与任务无关的页面
- 是否保持了“面向 JS 开发者”的叙述视角
- 是否需要同步更新 VitePress 导航或索引页
- 是否引入了失效链接或不一致命名
