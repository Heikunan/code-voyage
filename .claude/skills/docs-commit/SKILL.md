# Docs Commit

你是一个专门用于“提交当前文档站改动”的技能。

当前仓库是一个基于 VitePress 的技术学习文档站，内容主要围绕 Python、Rust、Solidity 学习路线，面向有 JavaScript / TypeScript 基础的开发者。

你的目标不是做通用 git 提交，而是：

1. 从“文档内容 / 导航结构 / 站点配置”视角理解改动
2. 生成适合该仓库语境的 commit message
3. 只提交与本次文档任务相关的文件
4. 在需要时提醒用户检查导航、索引和构建影响

## 适用场景

当用户提出以下需求时使用本技能：

- 提交这次文档改动
- 帮我提交 docs
- 给这次文档修改生成 commit message
- 把 VitePress 改动提交一下
- 提交学习路线/首页/侧边栏相关修改

## 仓库语境

提交前，始终把改动归类到以下一种或多种类型：

- 学习内容更新：某篇教程、章节、路线内容新增或补充
- 结构入口更新：`index.md`、目录页、阶段页、章节页入口调整
- 站点导航更新：`docs/.vitepress/config.ts` 中的 nav 或 sidebar 变更
- 站点配置更新：VitePress 配置、插件、部署或构建相关调整
- 首页或项目说明更新：`docs/index.md`、`README.md` 等站点级说明修改
- 文案修正：错别字、措辞、格式、小范围结构修复

## 默认工作流程

### 1. 先检查改动

查看：

- `git status --short`
- `git diff --staged`
- `git diff`
- 最近若干条 commit message

重点识别：

- 改动属于内容更新、导航更新、配置更新，还是文案修正
- 是否新增了页面
- 新增或重命名页面后，是否同步调整了导航、sidebar、索引页
- 是否同时修改了 `docs/.vitepress/config.ts`、`docs/index.md`、`README.md`
- 是否夹杂了与本次文档任务无关的文件

### 2. 按文档站语义总结提交意图

不要只按文件名机械描述，要总结“这次文档改动在站点中的意义”。

例如：

- 补充某条学习路线中的某个章节
- 更新首页入口以反映新的学习结构
- 为某个语言章节增加导航入口
- 调整 VitePress 配置以支持新的站点能力
- 修正文档中的错误表述或失效链接

## commit message 生成规则

优先沿用仓库现有风格，如：

- `docs: ...`
- `feat: ...`
- `fix: ...`
- `chore: ...`

### 推荐判断方式

优先按下面规则生成：

- 纯文档内容、说明、教程、章节补充：优先 `docs:`
- 为站点增加新的能力或用户可感知的新功能：可用 `feat:`
- 修复导航、链接、构建、配置、渲染问题：优先 `fix:`
- 维护性配置、小型工程调整、非用户内容层改动：优先 `chore:`

### 标题写法要求

- 简洁明确
- 要体现“改动对象 + 改动意图”
- 不要写成泛泛的 `update docs`
- 优先写出具体主题，例如：
  - `docs: add Rust ownership overview`
  - `docs: update Python phase index links`
  - `fix: correct VitePress sidebar paths`
  - `feat: add Mermaid support for VitePress docs`

如果仓库最近提交主要使用中文，也可以生成中文；否则优先跟随现有提交语言。

## 文件选择原则

- 只暂存和本次文档任务相关的文件
- 如果发现同时有代码、配置、草稿、临时文件混在一起，要先指出
- 不要擅自把无关 `.md`、截图、产物文件一起提交
- 如果改动涉及新页面，注意它的配套入口文件是否也应纳入本次提交

## 文档仓库专项检查

在真正提交前，尽量额外判断：

- 新文档是否放在合理目录
- 是否需要更新 `docs/.vitepress/config.ts`
- 是否需要更新对应 `index.md`
- 是否存在内部链接失效风险
- 如果属于结构性改动，是否应提醒用户执行 `npm run build`

如果只是非常小的文案修正，不要夸大检查范围。

## 输出给用户的方式

默认用简洁中文输出。

### 提交前

先简要告诉用户：

1. 这次改动被判断为什么类型
2. 准备提交哪些文件
3. 建议的 commit message
4. 如果发现导航/索引/配置遗漏风险，要明确说明

如果用户已经明确要求“直接提交”，且没有明显风险，可以直接执行。

### 提交后

告诉用户：

- 已创建 commit
- 最终 commit message
- 是否还有未提交改动
- 如果这是结构性文档改动，可补一句是否建议后续执行构建检查

## 特殊情况

### 情况 1：只有文案小修

优先生成简洁的 `docs:` 或 `fix:` 提交，不要夸大为功能提交。

### 情况 2：新增页面但没更新导航

先指出风险，再让用户决定：

- 是否本次一并补导航
- 还是先只提交页面内容

不要默认替用户扩大修改范围。

### 情况 3：同时改了内容和站点配置

可以合并成一次提交，但前提是它们服务于同一目标。

如果目标不同，应建议拆分提交。

### 情况 4：没有改动

直接说明当前没有可提交内容，不要创建空提交。

## 质量自检

提交前确认：

- 我是否看过 status 和 diff
- 我是否按文档站语义理解了这次修改
- commit message 是否足够具体
- 是否避免了把无关文件一起提交
- 是否识别了导航、索引、链接、配置相关风险
- 如果用户没有明确要求 commit，我是否仍停留在建议阶段
